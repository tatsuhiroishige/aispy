## 3. 既知の壁（未解決）

### 3.1 Inline 配置の崩壊

`<a href="..."><img ...></a>` のような inline wrapper 内の block-level image は IFC (`flattenInlines`) に入って alt text にフォールバックする。CSS 的には "block-in-inline split" で解決するが未実装。

現状 pipeline:
- `<img>` は `display: block` と判定されるが、inline 親（`<a>`）に囲まれているので layout は親の IFC 経路に入る
- IFC は image を atom として emit するが、tokenize で CJK alt text が 1 char 単位に splitされる（width=2 × N atoms）
- paint で image atom を検出しても、複数画像を 1 行の col 0 に prepend するので overlap して 1 枚しか見えない

### 3.2 Rate limit による decode 失敗

Wikipedia の `upload.wikimedia.org` は同時 request が 5 超えると 429 を返すことがある。concurrency=4 にしたが、97 img を順に fetch するので完全ではない。実測で `urlStore.size=35`（約 1/3 が成功）。

対策候補:
- retry with exponential backoff
- User-Agent 変更
- 画像 quality filter（alt が空、1x1px、favicon 除外）
- しばらく sleep を挟む

### 3.3 Kitty の `C=1` での複数画像重なり

`C=1` = cursor を動かさない。複数 APC を同じ line の先頭に prepend すると全部 col 0 に描画される。**Kitty Unicode Placeholders** を使えば解決するが、仕様が複雑（U+10EEEE + 列/行の combining diacritics + 24-bit RGB で image ID encode）。

対策候補:
- Kitty Unicode Placeholders protocol の実装
- OR: 画像は 1 行 1 枚に制限（現状 gallery mode）

### 3.4 Readability 依存

`htmlToText.ts` が Readability で main content 抽出してから render している。サイドバーの画像が落ちる場合あり。test-image コマンドは Readability を bypass するので raw HTML を処理。

## 5. 次 session の候補タスク

impact × effort 順:

1. **Inline image の本格対応** — Kitty Unicode Placeholders protocol + IFC の image atom 位置制御。一番インパクト大
2. **Rate limit 対応** — retry with backoff、User-Agent 調整。decode 成功率を 90%+ に
3. **Phase 4 Interactive browser mode** — URL bar / navigate / back/forward / tab（pivot の本来の次の一歩）
4. **Debug log 削除** — polish として最後に
5. **Block-in-inline split 実装** — IFC fallback の根本解決

### どこから再開すべきか

Gallery mode で画像描画パイプラインは動いているので、次は **Phase 4 (URL bar + navigate)** に進むか **3.7 polish（inline 配置）** に進むか。

- Phase 4 は pivot の本来の次の節目で、TUI browser として使い始められる
- 3.7 polish は画像体験を完成させる

私の推奨: **Phase 4 に進む**。画像の inline 配置は問題が複雑で、Phase 4 完了後に落ち着いて取り組む方が筋がいい。画像は gallery で"動いてる"扱いにして、先に browser 操作を完成させる。
