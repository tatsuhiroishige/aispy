### 1.3 2026-04-16 夕 session の追加

- **MCP fetch streaming** — `fetch-start → fetch → fetch-update × N` を IPC で段階配信。AI は single markdown response を受けるが、TUI は AI fetch でも画像 batch streaming で段階更新される。

- **Ctrl-R history search** — 全 tab の履歴 + bookmarks を fuzzy 検索する modal。↑↓/Enter で navigate。
- **Form input (MVP)** — `<form>` / `<input>` / `<textarea>` を parse、`F` で form modal、Tab で field 移動、Enter で submit。GET は query string、POST は `application/x-www-form-urlencoded` で axios.post。

### 2.3 Phase 3.7 polish

- **Rate limit 対策** (`imageDecoder.ts`): retry + exponential backoff + `Retry-After` 尊重、429/5xx/timeout で最大 3 回リトライ、User-Agent を Mozilla-compatible に
- **Block-in-inline split** (`anonymousBoxes.ts`): `<a><img></a>` で `<img>` (block display) が inline 親に閉じ込められる問題を根治。inline を block 前後で fragment に分割して block を hoist

### 2.4 MCP protocol 整理（2026-04-16 朝）

User の指摘で判明した 3 つの bug を修正:

**Bug 1: AI が Kitty escape まみれの response を受け取ってた**
- `fetch.ts` が `htmlToText(..., renderer: 'terminal')` を AI/TUI 両方に使ってた
- 修正: `htmlToMarkdown` (AI 用) + `htmlToTerminal` (TUI 用) を分離。AI には clean markdown、TUI には terminal render

**Bug 2: 50k chars truncation が TUI にも漏れてた**
- 修正: `fetchCache` を `{ content, aiContent, imagePrologue, tokens }` に拡張。TUI は full content、AI は truncated content

**Bug 3: CJK サイトで Claude Code の token 上限超過**
- 修正: char ベースじゃなく token ベース truncation。CJK=1.5 tokens/char、ASCII=0.25 tokens/char の推定で binary-search。上限 20k tokens

### 2.5 画像 prologue 分離（2026-04-16 夜）

Ink の `wrap-ansi` は CSI (color) は通すが **APC** (Kitty graphics) は壊す。content に混ぜると width 計算 / stripping で破綻し **画像が表示されない**。

- `renderHtmlToTerminalParts(doc, width) → { body, prologue }` に分離
- `HistoryEntry` / `FetchEvent` / `CachedPage` に `imagePrologue?` 追加
- `PageViewer` の `useEffect` で prologue を `process.stdout.write` (Ink 経由しない)、prologue 変わった時だけ re-emit

### 2.6 Option 2: 画像 streaming

```ts
async function* renderHtmlToTerminalStream(doc, width) {
  // Pass 1: 画像 prefetch なし → text + alt 描画 → 即 yield
  yield { body, prologue: '', phase: 'text' };
  // Pass 2: 画像 prefetch → 再 layout + paint → yield
  yield { body, prologue, phase: 'final' };
}
```

navigator → BrowserContext がストリーミング消費:

| yield | tab への適用 |
|---|---|
| Phase 1 (text) | `pushHistoryEntry` で新 entry commit、`loading=true` |
| Phase 2 (final) | `updateCurrentEntry` で同じ entry を上書き、`loading=false` |

履歴は **1 entry のまま**（Phase 2 で履歴重複しない）。

体感:
```
[~1s]  テキスト + [image: alt] 表示、ユーザーは読み始められる
[~5-10s] 画像が埋まって完成
```

Ghostty で Wikipedia 札幌記事 (19 images) で動作確認済（screenshot ベース）。

## 3. 主要アーキテクチャ

### 3.1 人間 navigation path

```
[g] URL input → navigator.navigateStream(url)
  → axios.get(html)
  → htmlToTerminalStream(html)      ← AsyncGenerator<TerminalContentUpdate>
      → Pass 1: build boxTree + layout + paint (no image)
      → yield { body, prologue: '', phase: 'text' }
      → Pass 2: prefetchImages + re-paint
      → yield { body, prologue, phase: 'final' }
  → BrowserContext: pushHistoryEntry (first) / updateCurrentEntry (rest)
  → PageViewer renders body via Ink, writes prologue to stdout
```

### 3.2 AI MCP fetch path

```
MCP fetch tool call
  → axios.get(html)
  → htmlToMarkdown(html)    ← clean markdown (AI 用)
  → htmlToTerminal(html)    ← terminal render (TUI 用)
  → applyAiTruncation(markdown, 20k tokens)
  → return { type: 'text', text: truncatedMarkdown } to AI
  → IPC: send { type: 'fetch', content: body, imagePrologue: prologue } to TUI
  → TUI: addFetchedTab → 新 tab で表示
```

## 5. 既知の壁 / 未対応

### 5.1 Phase 2 (text) → Phase 2 (final) の layout shift

画像が text pass では `[image: alt]` 幅 1 行、final pass では実 pixel cell size。layout がシフトする。

**対策案**: `<img width/height>` 属性を使って初期 layout を安定化。実装未着手（次 session 候補）。

### 5.2 画像が一斉に出現

現状 Phase 2 は 1 batch。全画像の prefetch 完了待ち。

**対策案**: batch streaming で 4 枚ごとに yield → 段階的に出現。実装未着手。

### 5.3 MCP fetch は streaming 非対応

MCP protocol が single response。TUI 側だけなら `fetch-update` event で段階更新できるが未実装。

### 5.4 大サイト AI fetch の token 上限

20k tokens で truncate。Claude Code の file-fallback に任せる選択肢もあるが、現状は truncate。

**対策案**: `section` / `offset` param で chunked retrieval。

### 5.5 iTerm2 での画像

OSC 1337 経路を維持してるが、複数画像で重なる可能性（Unicode Placeholders は Kitty/Ghostty/WezTerm のみ）。

### 5.6 Block-in-inline split の深い nest

`<a><span><img></span></a>` のような 2-depth 以上の inline 親は `hasBlockDescendant` で再帰 split するが、特殊 case（float 混在など）は未検証。
